            Environment Setup

            In this lab you will create 5 EC2 instances and publish the compiled application to the nodes. You will configure the systems in preparation for the subsequent labs and give yourself the tools you'll need to carry out the labs.

            The application and system configuration have been designed to demonstrate interesting problems in a safe environment. The intent of these labs are to demonstrate techniques for evaluating and improving the resilience of a distributed system, not to create a realistic production system.

            The goal of the application in this workshop is to demonstrate interesting behavior, so the system is intentionally not very resilient, especially in its initial state before adding mitigations. Do not use this application code in production.

            Environment Overview

            In these labs you will experiment on a chain of identical services. Each service has a single 'forward' operation which accepts an HTTP request and makes a request of the next service. You will use a load generator to send traffic into Service 1 and, later, use AWS Systems Manager to inject failures into Service 4.

            The graph below highlights how the load generator sends an HTTP request to Service 1. To process the request from the Load Generator, Service 1 sends a request to Service 2. Service 2 in turn sends a request to Service 3, which sends a request to Service 4. Service 4 is a CPU-bound process that performs useless calculations before responding to Service 3. Service 3 then returns this response to Service 2, which returns to Service 1, which then returns the complete response to the Load Generator.

            Every service will run on its own EC2 instance with the load generator running on the 5th instance. All instances should be in the same AWS region although they could be within different availability zones.

            Metrics

            Each of the 4 services will print metrics to its terminal window. As a result the terminal that is running a service will be your dashboard. Services report metrics every 4 seconds regardless of whether they are receiving traffic or not.

            The metrics produced are as follows:

            MetricDescription
            LatencyThe amount of time spent on each operation with values for mean, P50, P90, and P100. This is just the time spent in the service activity, it does not include the time a request spends in transit or waiting to be processed.
            ConnectionsThe average number of connections received per second during the reporting interval.
            ThroughputThe number of transactions processed per second during the reporting interval.
            Service AvailabilityThe percentage of requests that recieved a successful HTTP 200 response. This is defined as the number of successful transactions divided by the total number of transactions.
            Client AvailabilityThe percentage of requests which received a response before the client disconnected. This is defined as the number of successful transactions completed before the client times out divided by the total number of transactions.
            Quality of ServiceA number between 0 and 1 that tells what fraction of work is completed by the last service. The QoS number is passed back up the chain, so all the services will report the same QoS value.
            CPU UtilizationNumber between 0 and 1, that describes CPU utilization.

            Please note that availability numbers may fluctuate between 98% and 100% periodically. This is considered normal as calculations of availability will sometimes be performed while requests are in flight, leading to mismatched counts.
